Build DS-SOP Docker Image & Verify Components

The DS-SOP image wraps the DeepStream SOP pipeline with RTSP streaming output for VIOS integration. Its source is **generated by the ds-sop-skills skill** into `ds_sop_microservice`. RTSP streaming output is an **optional, opt-in** ds-sop-skills feature (§ 18) that this image requires, so the generation prompt must request it explicitly (*"with rtsp streaming output feature"*, § 2.1). vss-sop-build then **builds the source as-is — there is no patch step**.

`<workspace>` = working directory for SOP repos (e.g., `~/sop`).
`<bp-repo>` = path to `vss-sop` (e.g., `~/sop/vss-sop`).
`<ds-sop-src>` = ds-sop-skills generated source (default `<workspace>/ds_sop_microservice`).

## 2.1 Generate the DeepStream SOP source code (ds-sop-skills)

If `ds_sop_microservice` does not already exist, generate it with the ds-sop-skills skill. RTSP
output is opt-in (ds-sop-skills § 18), so the prompt **must** include *"with rtsp streaming output feature"*:

```
Please follow instructions in ds-sop-skills/example_sop_prompt.md to generate a SOP microservice with rtsp streaming output feature in folder ds_sop_microservice
```

This produces the DeepStream SOP microservice (FastAPI server, GEBD pipeline, VLM classification, SOP checker, Kafka/JSON messaging, Basler camera support) **with RTSP streaming output built in** (ds-sop-skills § 18) — standard layout: `docker/Docker.build` (GStreamer RTSP libs + codec plugins), `deploy/compose.yaml` (`ENABLE_RTSP_OUTPUT`/`RTSP_PORT`/`SW_ENCODER`), and `nvds_action_detector/{api_server,ds_3d_action_pipeline,ds_sop_process}.py` (`RTSPStreamingServer`, `tee1` output branch, `--rtsp-port`/auto-injection).

## 2.2 Build the image

`build_ds_sop.sh` validates the generated source, creates the `binaries/` bind-mount target, and builds `ds-sop:1.0.0` with BuildKit. It **does not clone and does not patch** anything — the generated source is built as-is.

```bash
cd <bp-repo>
./agentic/vss-sop-skills/vss-sop-build/scripts/build_ds_sop.sh <ds-sop-src> .
# <ds-sop-src> defaults to ../ds_sop_microservice relative to <bp-repo>
```

> The image is tagged `ds-sop:1.0.0`; the container workdir stays `/opt/nvidia/nvds_sop` (as generated). The VSS `ds-sop-docker-compose.yml` points `PYTHONPATH`/`working_dir`/mounts at `/opt/nvidia/nvds_sop`. See [`references/ds-sop-building.md`](../../vss-sop-build/references/ds-sop-building.md) for full details.

To run the build manually instead of via the script:

```bash
cd <ds-sop-src>
mkdir -p binaries

# Build using buildx to natively enable BuildKit (required for the optional binaries/ bind mount)
docker buildx build . -f docker/Docker.build -t ds-sop:1.0.0 --load
# Fallback if buildx is not installed:
# DOCKER_BUILDKIT=1 docker build . -f docker/Docker.build -t ds-sop:1.0.0
```

## 2.3 Verify the image exists

```bash
docker images ds-sop:1.0.0 --format '{{.Repository}}:{{.Tag}} {{.Size}}'
```

## 2.4 Verify RTSP Streaming Components

A suite of 6 checks must be performed on the built container to ensure the RTSP server packages, GStreamer elements, Python bindings, Protobuf stubs, and RTSP server instantiation are working properly.

Run the verification script directly from the host (it automatically spawns a container, mounts itself, and runs the diagnostics):

```bash
cd <bp-repo>
python3 agentic/vss-sop-skills/vss-sop-deploy/scripts/verify_rtsp_components.py
```

The script will verify:
1. `GstRtspServer` is importable.
2. `x264enc` software encoder is registered.
3. Protobuf stubs are successfully compiled.
4. All required GStreamer elements are present.
5. No broken shared library dependencies exist in the plugins.
6. `RTSPStreamingServer` can be instantiated and bound to a port.

> Note: `verify_rtsp_components.py` covers the RTSP/GStreamer components only (these 6 checks). The vss-sop-build skill's `post_build_checks.sh` runs these same 6 checks **plus** 2 additional model-wiring checks (Qwen3-VL detection and DDM-Net-only chunking) for 8 total — see [`ds-sop-building.md`](../../vss-sop-build/references/ds-sop-building.md). Use either script; they share Checks 1–6.
