# DS-SOP: Build the Generated SOP Microservice into the DS-SOP Image

Build the `ds-sop:1.0.0` Docker image directly from the SOP microservice source code **generated by the ds-sop-skills skill** into the `ds_sop_microservice` folder. **There is no patch step.**

> **Source & RTSP.** The source is **generated** by ds-sop-skills into `ds_sop_microservice`. RTSP streaming output is an **optional, opt-in** ds-sop-skills feature (§ 18); the VSS-SOP blueprint requires it, so the generation prompt must request it explicitly (*"with rtsp streaming output feature"*, see Step 1). `vss-sop-build` does **not modify the generated source** — it just builds it.

The image is **tagged** `ds-sop:1.0.0` at build time; the container's internal workdir stays `/opt/nvidia/nvds_sop` (as generated). The VSS `ds-sop-docker-compose.yml` points `PYTHONPATH`, `working_dir`, and the cache/chunks/alert volume mounts at `/opt/nvidia/nvds_sop` to match.

## Prerequisites

| Requirement | Detail |
|---|---|
| Base source | `ds_sop_microservice/` generated by the **ds-sop-skills** skill, with RTSP output opted in at generation (see Step 1) |
| Build script | `scripts/build_ds_sop.sh` (validate + build, no source modification) |
| Docker | with BuildKit |
| GPU | NVIDIA GPU with DeepStream 8.0 support |

### Step 0 — Verify Models & Download Assets

Before generating the source, ensure the SOP models and configs are present. For optimal accuracy, you must retrain/fine-tune the models, which can be done using the SOP Training Blueprint. After training, move model and config to `/opt/models/...` and `/opt/sop/...` directories. Run the asset script to verify them and prepare the test assets:

```bash
cd <bp-repo>
./agentic/vss-sop-skills/vss-sop-deploy/scripts/download_assets.sh --bp-repo .
```

This verifies that the required trained model and config files exist under `/opt/models` and `/opt/sop`. If not, prompts the user to retrain/fine-tune the models using the SOP Training Blueprint. It also downloads and re-encodes the sample RTSP video to H.264 at 30 FPS, and prepares the cache/datalog directories. See **vss-sop-deploy** SKILL.md § Phase 1 for details.

### Step 1 — Generate the source with ds-sop-skills

Generate the microservice (if it does not already exist) by following the ds-sop-skills prompt.
RTSP streaming output is an **optional, opt-in** ds-sop-skills feature (§ 18), and the VSS-SOP
blueprint **requires** it — so the generation prompt **must explicitly request RTSP output**:

```
Please follow instructions in ds-sop-skills/example_sop_prompt.md to generate a SOP microservice with rtsp streaming output feature in folder ds_sop_microservice
```

This produces the standard DeepStream SOP layout **with** RTSP output (see ds-sop-skills § 18 for the full file-by-file breakdown). Without the *"with rtsp streaming output feature"* phrase the generated source will omit RTSP and the RTSP post-build checks will fail.

## Building the Image

The build is automated by the skill-level script `../scripts/build_ds_sop.sh`. **It neither clones nor patches anything** — it validates the ds-sop-skills generated source and builds it:

```bash
./agentic/vss-sop-skills/vss-sop-build/scripts/build_ds_sop.sh <ds_sop_microservice_dir> <bp_repo_path>
# <ds_sop_microservice_dir> defaults to ../ds_sop_microservice (relative to the bp repo)
# <bp_repo_path>            defaults to .
```

It performs:

1. Validate that `<ds_sop_microservice_dir>` exists and contains the expected DeepStream SOP layout (`docker/Docker.build`, `deploy/compose.yaml`, `nvds_action_detector/{api_server,ds_3d_action_pipeline,ds_sop_process}.py`). If missing, it errors and tells you to generate the source with the ds-sop-skills skill first.
2. Create the `binaries/` bind-mount target if absent.
3. Build `ds-sop:1.0.0` with `docker buildx build . -f docker/Docker.build -t ds-sop:1.0.0 --load` (or fallback: `DOCKER_BUILDKIT=1 docker build . -f docker/Docker.build -t ds-sop:1.0.0`).

## Post-Build Verification & Auto-Fix

Run the 8 checks **in order** against the built `ds-sop:1.0.0` image. Stop and fix before moving on.

Two scripts cover the full workflow:

- `./scripts/ds-sop/post_build_checks.sh` — runs all 8 checks sequentially. Use `--container` to launch a fresh container automatically (`docker run --rm --gpus all -v ...:/tmp/post_build_checks.sh:ro --entrypoint bash ds-sop:1.0.0 /tmp/post_build_checks.sh`); otherwise run inside an existing container shell.
- `./scripts/ds-sop/apply_fixes.sh [check2|check3|check4|check5|all]` — applies remediation for the 4 checks that have automatable fixes. Run **inside** the container; after fixes apply cleanly, port the changes back into `docker/Docker.build` and rebuild.

To enter a container shell manually for ad-hoc inspection:

    docker run --rm -it --gpus all --entrypoint bash ds-sop:1.0.0

### What each check validates

| # | Check | Validates | Auto-fix |
|---|---|---|---|
| 1 | `GstRtspServer` importable | RTSP server typelib (`libgstrtspserver-1.0-dev`, `gir1.2-gst-rtsp-server-1.0`) | manual: add packages to Dockerfile |
| 2 | `x264enc` available | Software H.264 encoder plugin registered | `apply_fixes.sh check2` (install `gstreamer1.0-plugins-{ugly,bad}` + `libav`, rebuild registry) |
| 3 | Protobuf stubs compiled | `nvds_action_detector/protos/{nv,ext}_pb2.py` exist with descriptors | `apply_fixes.sh check3` (`protoc -I. --python_out=. nv.proto ext.proto`; rewrite `ext_pb2.py` import) |
| 4 | GStreamer registry plugins | `x264enc`, `rtph264pay`, `udpsink`, `nvvideoconvert`, `jpegenc`, `rtpjpegpay` all findable | `apply_fixes.sh check4` (clear cache + re-inspect; add packages: `x264enc`→ugly, `jpegenc/rtph264pay/rtpjpegpay/udpsink`→good, `nvvideoconvert`→deepstream) |
| 5 | Shared library deps satisfied | No `not found` in `ldd` of `gstreamer-1.0/libgst*.so` (codec plugins fail silently otherwise) | `apply_fixes.sh check5` (reinstall `libvpx9 libzvbi0t64 libmp3lame0 libx265-199 libunibreak5 libmpg123-0t64`) |
| 6 | `RTSPStreamingServer` instantiable | End-to-end RTSP server attaches to a GLib MainContext | manual: re-check Checks 1 and 4 (typically missing `GstRtspServer` typelib) |
| 7 | Qwen3-VL detection wired | `_is_qwen3vl` and `return_video_metadata` present in `VLLMInference` source | manual: regenerate with ds-sop-skills (the generated `vllm_inference.py` should already wire this) |
| 8 | Chunking = DDM-Net only | `UniformChunkingOptions` removed from `ChunkingOptions` union | manual: regenerate with ds-sop-skills (the generated `api_types.py` should already exclude it) |

All 8 must pass. After auto-fixing inside the container, update `docker/Docker.build` and rebuild the image. If checks 7–8 fail, the fix lives in the **ds-sop-skills** generation (not in `vss-sop-build`) — regenerate `ds_sop_microservice`.

> Note: Checks 1–6 are identical to the RTSP component checks in `verify_rtsp_components.py` (the 6-check script used by the vss-sop-deploy flow — see [`build_ds_sop_image.md`](../../../vss-sop-deploy/references/build_ds_sop_image.md)). `post_build_checks.sh` adds Checks 7–8 to also validate the Qwen3-VL and chunking behavior, for 8 total. All of these now validate the **generated + built** image rather than a downstream patch.

## Runtime Configuration

| Env var | Default | Description |
|---|---|---|
| `ENABLE_RTSP_OUTPUT` | `true` | Auto-enable RTSP output for live (`rtsp://` / `camera://`) inputs |
| `SW_ENCODER` | `true` | CPU `x264enc` instead of GPU `nvv4l2h264enc` |
| `RTSP_PORT` | `8554` | RTSP server listening port |

### CLI usage

Two entry points exist inside the container; both accept `--video-path` and `--rtsp-port`:

- Standalone pipeline: `python -m nvds_action_detector.ds_3d_action_pipeline --video-path /path/to/video.mp4 --rtsp-port 8554` → `rtsp://127.0.0.1:8554/ds-test`
- SOP process: `python -m nvds_action_detector.ds_sop_process --video-path /path/to/video.mp4 --rtsp-port 8554` → `rtsp://127.0.0.1:8554/ds-out/{video_name}`

The API server reads `RTSP_PORT` from env and auto-configures RTSP output for RTSP/Basler inputs at `/ds-out/{sensor_id}`.

### Viewing the RTSP stream

Three players work out-of-the-box:

- `ffplay rtsp://127.0.0.1:8554/ds-out/my_camera`
- `vlc rtsp://127.0.0.1:8554/ds-out/my_camera`
- GStreamer: `gst-launch-1.0 rtspsrc location=rtsp://127.0.0.1:8554/ds-out/my_camera latency=0 ! rtph264depay ! h264parse ! avdec_h264 ! videoconvert ! autovideosink`

## Integration Troubleshooting

### Unsupported file type error with VST direct downloads
If you see the following error when `ds-sop` downloads video clips from VST directly:
```
MediaInfo.parse: Unsupported file type
```
This is because direct downloads of GStreamer-created container files can occasionally hit format/mime-type parsing issues in `MediaInfo`.

**Resolution:**
Force the VSS Agent to send base64-encoded videos instead of VST links. This is done by adding/ensuring `use_base64: true` under the `video_understanding` block in `configs/config.yml`:

```yaml
functions:
  video_understanding:
    _type: video_understanding
    ...
    use_base64: true
```
